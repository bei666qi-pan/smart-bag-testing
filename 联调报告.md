## 二、ESP32 固件改了什么

### 1. 屏幕全链路（此前几乎全断）

| 缺陷 | 现象 | 修复 |
| --- | --- | --- |
| `Screen_setup()` 在 `.ino` 里写成 `void Screen_setup();`（一条函数声明语句，等于空操作） | 屏幕**从未真正初始化**，所有上屏无效 | 改为真正调用 `Screen_setup();` |
| `handleScreenEvents()` 从未在 `loop()` 调用 | 屏幕按钮 b0/b1/b2（开灯/关灯/拍照）**完全失灵** | 在 `loop()` 中每轮调用 |
| `screen_text` 命令分支是 `// TODO`，只打印不显示 | 家长下发的文字**到不了屏幕** | 调用 `updateWritingText()` + `updateAIResponse()` 真正上屏 |
| 拍照按钮调用 `camera_system_loop()`（受 10s 限速，按下不一定真拍） | 按拍照按钮经常无反应 | 新增 `camera_system_capture_now()`，立即抓拍并上传一次 |
| 缺 GPS 上屏、AI 回复上屏接口 | 屏幕信息不全 | 新增 `updateGps()`，主循环每轮把经纬度刷到 `t_gps` |

### 2. MQTT 命令闭环

- `screen_text` / `mode_switch` 命令现在**真正驱动屏幕与灯效**（之前是 TODO / 仅打印）。
- ACK 回执字段确认为 **`cmd_id`**（网页端用它匹配 `pendingCmd`，契约要求）；
  未识别 action / 非法 mode 时回 `status=1`，便于网页端判断命令失败。
- `screen_text` 命令较长时不再被截断：`client.setBufferSize(1024)`。
- 修复**传感器异常值过滤逻辑**：原判据 `if (temp < -40 && temp > 85 && humid <= 0 && humid >= 100)`
  恒为假（永不过滤），改为正确的越界判据 `temp<-40 || temp>85 || humid<0 || humid>100`，
  且读取失败也输出 9999 哨兵值，避免上报陈旧数据。

### 3. UART 硬件串口冲突（关键隐患）

- 原代码屏幕用 `Serial2`、GPS（NEO_M9N）用 `HardwareSerial(2)` —— **同占 UART2，必然冲突**。
- 修复：屏幕迁到 **UART1**（独立 `HardwareSerial ScreenSerial(1)`，引脚仍为 GPIO42/48），
  GPS 保留 **UART2**。二者彻底分离。
- 顺带修复 `NEO_M9N_Init()` 里 `Serial.begin(38400)` 把调试串口波特率从 115200 改成 38400
  导致日志乱码的问题；`MQTT_set()` 重复 `Serial.begin` 移除，并给 WiFi 连接加 20s 超时
  （无热点时不再永久卡死）。

### 改动文件清单

```
esp32代码/Smart_Schoolbag_03.ino      主循环修复（Screen_setup 调用、handleScreenEvents、过滤逻辑、GPS上屏）
esp32代码/MQTT.cpp                     命令闭环、ACK、缓冲区、WiFi 超时、串口修复
esp32代码/Hardware/Screen.h           UART1 配置 + 新增 updateGps 声明
esp32代码/Hardware/Screen.cpp         屏幕模块重写（独立 UART1、screen_text/AI/GPS 上屏、立即拍照）
esp32代码/Hardware/camera.h           新增 camera_system_capture_now() 声明（移除头里的 static 前置声明）
esp32代码/Hardware/camera.cpp         新增 camera_system_capture_now() 实现
esp32代码/Hardware/NEO_M9N.cpp        去掉会污染调试串口波特率的 Serial.begin(38400)
```

## 三、验证方法与结果

### 1. 固件编译验证

由于沙箱无法下载 ESP32 工具链（espressif / arduino 下载域被出网策略屏蔽），
采用「真实 g++ + Arduino/ESP32 API 桩」对固件**自身逻辑**做编译级验证：

- `g++ -fsyntax-only`：**0 错误**（类型、签名、声明全部自洽）
- `g++` 完整编译 + 链接：**0 未定义符号**（每个声明的函数都有定义，含新增的
  `camera_system_capture_now`、各屏幕函数、MQTT 回调）
- 运行 `setup()/loop()` 一轮：**正常退出，无逻辑崩溃**
- 命令路径残留 TODO 扫描：**0 条**

> 局限：API 是桩实现，不等于真机固件烧录通过。但本次问题集中在
> 「函数没被调用 / 声明与定义不匹配 / 串口冲突」这一类，正是编译+链接能稳定捕获的。
> 接真机时用 Arduino IDE / arduino-cli 选 ESP32-S3 板再编一次即可。

### 2. 软件全链路联调（真实 MQTT 协议，8/8 通过）

测试拓扑（全部在一个进程内，秒级跑完）：

```
模拟ESP32(复刻固件协议) ──MQTT──> 本地真实 broker(amqtt) ──> 模拟服务端daemon ──> 模拟网页服务端(/api/iot/state)
       │                                                                              ▲
       └────────── HTTP 多部分表单 + x-device-token ───────> /api/camera/latest ──────┘
模拟网页端 ──下发 v5/bag/cmd──> broker ──> 模拟ESP32 ──回 v5/bag/cmd/ack(cmd_id)──> 模拟网页端
```

测试输出：

```
[PASS] 步骤1 state接口可达(daemon已起)
[PASS] 步骤2 设备 status=online 镜像 -> status=online
[PASS] 步骤3 sensors 镜像 temp=24.5 humid=45.0 battery=88
[PASS] 步骤4 gps 镜像 lat=31.230412 lng=121.473701
[PASS] 附加 摄像头快照上传->拉取(image/jpeg)
[PASS] 步骤5 screen_text 驱动屏幕 t_writing/t_replay  t_writing='记得带作业本'
[PASS] 步骤6 设备回 cmd/ack 且 cmd_id 匹配(命令闭环)  ack={cmd_id:..., status:0, msg:OK}
[PASS] 附加 mode_switch focus_mode 驱动呼吸蓝灯+ACK
=== 结果: 8/8 通过 ===（重复运行稳定）
```

覆盖到的契约点：设备在线判定只认 `v5/bag/status`；ACK 字段必须叫 `cmd_id`；
摄像头走 `x-device-token: <DEVICE_TOKEN>` 多部分表单上传、`image` 字段名；
`/api/iot/state` 镜像字段；`screen_text` / `mode_switch(focus/normal)` 命令语义。

## 四、接真机时怎么用这套测试

`联调测试/selftest_oneshot.py` 是一体化自测（自带 broker，零外部依赖，`pip install paho-mqtt amqtt` 即可跑）。

若要对**真实生产链路**联调，改用 `联调测试/` 下的拆分脚本，把地址换成生产值：

1. ESP32 真机烧录修复后的固件，开机上电。
2. `python3 daemon_mirror.py mqtt.bag.versecraft.cn 1883 https://bag.versecraft.cn` 起镜像。
3. 浏览器开 `/interaction`，依次确认：设备在线 → 传感器/GPS 实时刷新 → 下发 `screen_text`
   屏幕显示 → 收到 ACK；`/api/iot/daemon-status` 显示 `started=true`、`subscribed=true`。

达成对接文档第 8 节 6 步，即「软硬件主链路已打通」。

## 五、仍需真机确认的项（诚实声明）

以下逻辑已在代码与软件链路层面验证，但**只有接真机才能最终确认**：

- 串口屏控件名是否与屏幕工程一致（`t_temperature/t_humidity/t_light/t_writing/t_replay/t_gps`、
  课表 `class1_text`… 等）——若屏幕工程里控件名不同，需按实际名再改 `Screen.cpp`。
- UART1（GPIO42/48）在你的板子上是否被其它外设占用。
- AHT10 / NEO-M9N / OV 摄像头在实物上的初始化与读数。
- 生产 broker / WSS / Redis / NewAPI 的可达性与鉴权。
